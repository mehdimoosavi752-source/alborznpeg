# نوین پلی‌تکنیک — نسخه‌ی کامل با بک‌اند واقعی

این نسخه، برخلاف نسخه‌ی دموی قبلی، یک **بک‌اند واقعی** دارد:

- دیتابیس واقعی (SQLite) به‌جای localStorage مرورگر
- رمزهای عبور با **bcrypt هش** می‌شوند، نه متن ساده
- ورود با **JWT** انجام می‌شود
- تشخیص «مدیر بودن» واقعاً **سمت سرور** چک می‌شود، نه با یک متغیر قابل‌دستکاری در مرورگر
- هیچ‌کس با باز کردن Developer Tools نمی‌تواند خودش را مدیر کند

ساختار پوشه‌ها:
```
novin-fullstack/
├── server/   ← بک‌اند Node.js + Express + SQLite
└── client/   ← فرانت‌اند React + Vite (همان طراحی قبلی)
```

---

## ۱) اجرای محلی (برای تست قبل از دیپلوی)

### سرور
```bash
cd server
cp .env.example .env
# فایل .env را باز کنید و JWT_SECRET را با یک مقدار تصادفی طولانی جایگزین کنید
npm install
npm run dev
```
سرور روی `http://localhost:8787` بالا می‌آید و به‌صورت خودکار یک دیتابیس SQLite (`data.sqlite`) و یک حساب مدیر پیش‌فرض می‌سازد (کاربری `admin`، رمز `novin1404` — یا هرچه در `.env` گذاشتید).

### کلاینت
```bash
cd client
cp .env.example .env
npm install
npm run dev
```
حالا آدرسی که ترمینال نشان می‌دهد (معمولاً `http://localhost:5173`) را باز کنید. کلاینت به‌صورت خودکار به سرور روی `localhost:8787` وصل می‌شود.

## ۲) دیپلوی روی Render — فقط با یک سرویس

خبر خوب: سرور الان طوری تنظیم شده که خودش فایل‌های ساخته‌شده‌ی کلاینت را هم سرو می‌کند. یعنی دیگر لازم نیست دو سرویس جدا بسازید — همه‌چیز در **یک Web Service** بالا می‌آید.

1. کل پوشه‌ی `novin-fullstack` (شامل `client`، `server` و همین `package.json` ریشه) را در گیت‌هاب push کنید.
2. در Render: **New + → Web Service**
3. مخزن گیت‌هاب را وصل کنید
4. تنظیمات:
   - **Root Directory:** خالی بگذارید (ریشه‌ی مخزن)
   - **Build Command:** `npm run build`
   - **Start Command:** `npm start`
5. در بخش **Environment**، این متغیرها را اضافه کنید:
   - `JWT_SECRET` → یک رشته‌ی تصادفی طولانی (همین الان بسازید؛ مثلاً با اجرای `node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"` روی سیستم خودتان)
   - `ADMIN_USERNAME` → مثلاً `admin`
   - `ADMIN_PASSWORD` → یک رمز قوی دلخواه
6. Create Web Service را بزنید.

همین. چون فرانت‌اند و بک‌اند روی یک آدرس (origin) سرو می‌شوند، دیگر نیازی به تنظیم `CORS_ORIGIN` یا `VITE_API_URL` هم نیست — این‌ها فقط برای زمانی هستند که بخواهید کلاینت و سرور را در دو سرویس جدا نگه دارید (مثلاً برای مقیاس‌پذیری بیشتر در آینده؛ روش قبلی هنوز هم کار می‌کند، فقط دیگر پیش‌فرض نیست).

> ⚠️ **نکته‌ی مهم درباره‌ی ماندگاری داده روی Render:**
> پلن رایگان Render فضای دیسک را بین ری‌استارت‌ها/دیپلوی‌های جدید **پاک می‌کند**، یعنی فایل `data.sqlite` (و هرچه در پنل مدیریت اضافه کرده‌اید) از بین می‌رود. برای یک فروشگاه واقعی که داده‌هایش نباید پاک شود، یکی از این دو راه را انتخاب کنید:
> - از **Render Disks** استفاده کنید (فضای دیسک دائمی، پلن پولی) و متغیر `DB_PATH` را به مسیر دیسک متصل‌شده بدهید.
> - یا دیتابیس را از SQLite به یک دیتابیس مدیریت‌شده مثل **Render PostgreSQL** (پلن رایگان هم دارد) یا **Supabase** تغییر دهید. اگر بخواهید، می‌توانم همین الان این تغییر را هم برایتان انجام بدهم تا این نگرانی هم برطرف شود.

---

## ۳) بعد از دیپلوی چه کار کنم؟

1. با حساب مدیر (`ADMIN_USERNAME` / `ADMIN_PASSWORD` که در Environment سرور گذاشتید) وارد شوید
2. حتماً **رمز عبور پیش‌فرض را عوض کنید** — فعلاً امکان تغییر رمز از پنل وجود ندارد؛ اگر لازم دارید بگویید تا این قابلیت را هم اضافه کنم
3. محتوای سایت (خدمات، محصولات، صفحات، منو، فوتر) را از پنل مدیریت ویرایش کنید

## نکته درباره‌ی درگاه پرداخت

بخش پرداخت همچنان **شبیه‌سازی‌شده** است. برای اتصال به یک درگاه واقعی (زرین‌پال، به‌پرداخت ملت و...):
- باید در `server/index.js` دو endpoint اضافه شود: یکی برای شروع پرداخت (ارسال درخواست به درگاه و گرفتن لینک پرداخت) و یکی برای callback تایید پرداخت
- کلید API درگاه باید فقط در `.env` سرور باشد، هرگز در کد کلاینت
- این کار نیاز به داشتن یک حساب پذیرنده (merchant account) نزد درگاه بانکی دارد که باید خودتان ثبت کنید؛ وقتی اطلاعات پذیرندگی را داشتید، می‌توانم این اتصال را برایتان پیاده کنم.

## امنیت — چه چیزهایی الان درست شده

- ✅ رمزها هش می‌شوند (bcrypt)
- ✅ نقش مدیر سمت سرور چک می‌شود، نه در کد فرانت‌اند
- ✅ توکن‌های ورود منقضی می‌شوند (۷ روز)
- ✅ endpoint های حساس (ویرایش محتوا، مشاهده‌ی کاربران/سفارشات/پیام‌ها) نیازمند نقش مدیر هستند

## چیزهایی که هنوز باقی مانده (پیشنهاد برای آینده)

- Rate limiting روی endpoint های لاگین/ثبت‌نام (جلوگیری از حمله‌ی brute-force)
- قابلیت تغییر رمز عبور و فراموشی رمز (ایمیل/پیامک بازیابی)
- اعتبارسنجی دقیق‌تر ورودی‌ها (مثلاً با Zod)
- HTTPS و دامنه‌ی اختصاصی (سرویس‌های Render/Vercel به‌صورت پیش‌فرض HTTPS رایگان می‌دهند)
- لاگ‌گیری و مانیتورینگ خطاهای سرور
